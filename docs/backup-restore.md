## 1. バックアップの種類

バックアップには2種類あります。**カスタムネットワークを運用している場合はフルバックアップを強く推奨**します。

| | 設定のみバックアップ | フルバックアップ |
|---|---|---|
| ノード設定 (custom-preset.yml / ui-meta) | ✅ | ✅ |
| ノード鍵 (addresses.yml) | ✅ | ✅ |
| ジェネシスブロック (nemesis seed / transactions) | ✅ | ✅ |
| 委任ハーベスター (harvesters.dat) | ✅ | ✅ |
| **ブロックデータ** (nodes/*/data) | ❌ | ✅ |
| **MongoDB** (databases/) | ❌ | ✅ |
| ノード停止の要否 | 不要 | **必須** |
| ZIP サイズ | 数十 KB | チェーンサイズに依存(圧縮されます) |
| 復元後のチェーン | 下記「ケース別」参照 | **高さそのまま継続** |

### どちらを使うべきか

- **カスタムネットワーク(自分で構築したネットワーク)**
  → **フルバックアップ**。このノードのブロックデータがチェーンの唯一のコピーです。
  設定のみのバックアップから復元すると、チェーンはジェネシス(高さ1)から再構築され、
  **それまでのトランザクション履歴(送金・モザイク作成など)はすべて失われます**。
- **公式ネットワーク (mainnet / testnet) や既存ネットワークへの参加ノード**
  → 設定のみで十分です。ブロックデータはネットワークから再同期できます。
  (フルバックアップにすれば復元後の再同期時間を短縮できます)

## 2. バックアップの作成手順

1. サイドバーの **Backup** タブを開く
2. 「バックアップ対象ファイル」の一覧で各ファイルが「利用可能」になっていることを確認
3. **フルバックアップの場合**:
   - 先にダッシュボードまたは Operations ページで **Stop** を実行しノードを停止する
     (稼働中はチェックボックスが無効になり、理由が表示されます)
   - 「**フルバックアップ(ブロックデータを含む)**」にチェックを入れる
     (横にサイズ目安が表示されます)
4. **バックアップをダウンロード** ボタンを押す
   → `node-backup-<日時>.zip` または `node-full-backup-<日時>.zip` がダウンロードされます

### 保管時の注意

- ZIP には**ノードの秘密鍵 (addresses.yml) が含まれます**。第三者に渡らないよう安全な場所に保管してください
  (秘密鍵は起動時に指定した「ネットワークパスワード」で暗号化されています)
- 定期運用では、チェーンが進むたびにフルバックアップを取り直すことを推奨します
  (復元されるのは**バックアップ時点の高さまで**です)

## 3. リストア手順

### 前提条件

- ノードが**停止していること**(稼働中はアップロードが拒否されます)
- **バックアップ作成時と同じネットワークパスワード**を知っていること
  (addresses.yml の秘密鍵がこのパスワードで暗号化されているため、違うパスワードでは起動できません)
- 管理画面にログインしていること

### 手順

1. ノードが稼働中なら **Stop** で停止する
2. サイドバーの **Backup** タブで「リストア」セクションを開く
3. バックアップ ZIP をドロップゾーンにドラッグ&ドロップ(またはクリックしてファイル選択)
4. 確認ダイアログで内容を確認して OK
   ⚠️ **この時点で既存のノードデータ(ブロック・DB・生成済み設定)は削除されます**
5. アップロード進捗が 100% になり、完了メッセージが表示されるのを待つ
   - フルバックアップの場合: 「Full backup restored. Start the node to apply...」
   - ブロックデータは一時領域(`.pending-restore/`)に展開され、次回起動時に設置されます
6. **Start** ボタンでノードを起動する
   - パスワードは**バックアップ作成時と同じもの**を入力
7. ターミナルログで以下を確認:

```
[Restore] 📦 Installed databases/ (MongoDB restored)
[Restore] 📦 Installed block data → api-node-0/data (chain height preserved)
[Restore] 🌾 Installed harvesters.dat → api-node-0/data/ (delegators carried over)
```

### 復元後の確認ポイント

| 確認項目 | 場所 | 期待値 |
|---|---|---|
| ブロック高 | ダッシュボード | フル: バックアップ時の高さから継続 / 設定のみ: 1 から |
| ネットワーク通貨 | ダッシュボード | バックアップ時と同じ通貨名・モザイクID |
| ノード名 (friendlyName) | ダッシュボード | バックアップ時と同じ |
| ノードアドレス・公開鍵 | アドレスページ | バックアップ時と同じ(identity が維持されている証拠) |
| 委任者 | ハーベスト状況 | 委任ハーベスターが継続 |

## 4. ケース別の復元動作

### フルバックアップ → カスタムネットワーク

チェーンがバックアップ時点の高さのまま復元されます。ノード鍵・ジェネシス・
ブロックデータ・REST 用 DB・委任者すべてが引き継がれ、そのまま運用を継続できます。
**別マシンへの引っ越しにも使えます**(移行先で同じ ZIP をリストアするだけ)。

### 設定のみバックアップ → カスタムネットワーク

ノード identity とジェネシスは復元されますが、チェーンは**高さ1から再構築**されます。
- 他に同じネットワークのノードが稼働していれば、そこから再同期して復旧できます
- **単一ノード構成では過去のトランザクション履歴は失われます**(ジェネシス配布分の残高はやり直しで再現されます)

### 公式ネットワーク (mainnet / testnet) の参加ノード

ノード identity(鍵・証明書のもとになる addresses.yml)と設定が復元され、
ブロックは公式ネットワークから再同期されます。フルバックアップならバックアップ時点の
高さまでは同期済みの状態で始まり、差分だけ同期します。

### 委任ハーベスター (harvesters.dat) について

harvesters.dat の中身は**ノードの transport 鍵で暗号化**されています。
同じバックアップ内の addresses.yml とセットで復元されるため通常は問題ありませんが、
復元後に**フルリセットして鍵を再生成した場合は無効**になります。その場合、
委任者は委任ハーベスティングリクエストを再送する必要があります。

## 5. 注意事項

- **リストアは破壊的操作です**。実行前に、現在の状態が必要なら先にバックアップを取ってください
- リストア後の初回起動は「フル起動」(設定再生成)になるため、通常より時間がかかります
- Docker イメージ(symbol-server 等)はバックアップに含まれません。移行先が初回の場合、
  起動時に自動ダウンロード・ビルドされます(インターネット接続が必要)
- ネットワークパスワードを忘れた場合、バックアップからの復元はできません
  (秘密鍵を復号できないため)。パスワードは別途安全に管理してください
- 管理画面のログインセッションは24時間で切れます。アップロード前にセッション切れの
  アラートが出た場合は再ログインしてからやり直してください(操作は実行されていません)

## 6. トラブルシューティング

| 症状 | 原因と対処 |
|---|---|
| アップロードが 409 エラー | ノードが稼働中です。Stop してから再実行 |
| 「Invalid backup: custom-preset.yml not found」 | このツールで作成した ZIP ではありません。ファイルを確認 |
| フルバックアップのチェックが押せない | ノードが稼働中です。Stop すると有効になります |
| 復元後にブロック高が 1 に戻った | 使用した ZIP が「設定のみ」バックアップです。ターミナルログに `📦 Installed block data` が出ていたか確認 |
| 起動時に鍵関連のエラー | 入力したネットワークパスワードがバックアップ作成時と異なります |
| 委任者が消えた | ZIP に harvesters.dat が含まれていたか(バックアップ画面の一覧で「利用可能」だったか)確認。含まれていても、復元後にフルリセットすると無効になります |
| アップロードが途中で失敗する | ターミナルログの `[Restore]` 行を確認。ディスク空き容量(展開に ZIP サイズの2倍程度)も確認 |

ログはヘッダーの **Terminal Logs** で確認できます。リストア関連の行は `[Restore]` プレフィックス付きです。

## 7. 付録: バックアップ ZIP の構造

```
node-full-backup-2026-07-03T12-00-00.zip
├── backup-meta.json              # メタデータ (形式バージョン・種別・作成日時)
├── custom-preset.yml             # ノード/ネットワーク設定
├── ui-meta.json                  # UI メタデータ (プリセット種別・アセンブリ等)
├── addresses.yml                 # ノード鍵 (暗号化済み) ⚠️ 秘密情報
├── nemesis/
│   ├── seed/00000/…              # ジェネシスブロック
│   ├── seed/index.dat
│   └── transactions/…            # ジェネシストランザクション
├── harvesters/
│   └── <ノード名>/harvesters.dat  # 委任ハーベスター
├── blockdata/                    # ★フルバックアップのみ
│   └── <ノード名>/…               # ブロックデータ (data/ 一式)
└── databases/                    # ★フルバックアップのみ
    └── db/…                      # MongoDB (REST 用インデックス)
```

`.lock` ファイルや `server-recovery.started` などのランタイム一時ファイルは
バックアップ時に自動的に除外されます。
